# Arcana Push Protocol v1

Status: Draft for review

Scope: SillyTavern Arcana extension -> Arcana web reader.

This document defines the Phase 1 push protocol for replacing URL-hash transfer with a reusable Arcana browser window, `postMessage`, and explicit ACK confirmation.

## Goals

- Reuse one Arcana browser tab/window for repeated pushes.
- Stop sending large archives through `#push=...` URLs.
- Confirm that Arcana actually saved data before the extension updates push history.
- Keep the current `.archive.json` manual import fallback.
- Preserve the current content extraction pipeline in the extension.

## Non-Goals

- This protocol does not implement cloud sync.
- This protocol does not require an APK, local server, or backend API.
- This protocol does not redesign regex rendering or IndexedDB storage in Phase 1.
- This protocol does not remove the existing `#push=` receiver until the new path is proven stable.

## Roles

- Extension: the SillyTavern extension that collects chat data and sends pushes.
- Arcana Web: the reader page that receives pushes and writes data to IndexedDB.
- Push window: the reusable Arcana window returned by `window.open`.

## Window Reuse

The extension must open Arcana with a stable window name:

```js
window.open(archiveUrl, 'arcana-reader')
```

Rules:

- The window name is exactly `arcana-reader`.
- The extension should reuse the returned window reference while it remains open.
- The extension must not use `_blank` for normal Phase 1 pushes.
- If the stored window reference is missing or closed, the extension may call `window.open(archiveUrl, 'arcana-reader')` again.
- The extension generates one `channelId` when it opens or reuses an Arcana window. This `channelId` represents the current window connection session and remains stable while that window is considered alive.
- If the Arcana window is closed, replaced, or reopened after the extension has lost the previous window reference, the extension must generate a new `channelId`.
- If `window.open` returns `null`, the extension must treat this as popup-blocked and fall back to `.archive.json` download or show a clear failure message.

## Origin Policy

The extension must derive `targetOrigin` from the configured Arcana URL:

- `https://example.com/arcana/index.html` -> `https://example.com`
- `http://127.0.0.1:8080/index.html` -> `http://127.0.0.1:8080`

Rules:

- For `http:` and `https:` Arcana URLs, `postMessage` must use the exact `targetOrigin`.
- The extension must verify ACK messages by checking both `event.origin === targetOrigin` and `event.source === pushWindow`.
- Arcana Web should verify incoming messages by checking protocol fields, supported message types, and origin where available. It must not depend on `window.opener`, because mobile browsers may set cross-origin `window.opener` to `null`.
- If `archiveUrl` starts with `file://`, the extension must automatically use the legacy `#push=` path and must not attempt the Phase 1 `postMessage` protocol. No user choice is required for this fallback.
- Mobile browsers may silently drop messages when origin or timing is wrong. The extension must use ping/readiness polling and retry with explicit timeouts instead of assuming desktop behavior proves mobile behavior.

## Message Envelope

Every protocol message must include these envelope fields:

```js
{
  _protocol: 1,
  _messageType: 'arcana.push',
  _pushType: 'story.full',
  pushId: 'string',
  channelId: 'string',
  sentAt: '2026-07-01T00:00:00.000Z'
}
```

Envelope fields:

| Field | Required | Type | Meaning |
|---|---:|---|---|
| `_protocol` | yes | number | Protocol version. Phase 1 value is `1`. |
| `_messageType` | yes | string | Message category. Readiness probes use `arcana.ping`. Ready replies use `arcana.ready`. Push requests use `arcana.push`. ACK messages use `arcana.ack`. |
| `_pushType` | yes for push/ACK | string | Payload type. See push types below. |
| `pushId` | yes for push/ACK | string | Unique ID for one push attempt. Used for ACK matching and idempotency. |
| `channelId` | yes | string | Window connection session ID generated by the extension when it opens or reuses the Arcana window. Used to reject unrelated messages. |
| `sentAt` | yes for push | ISO string | When the extension sent this message. |

`pushId` requirements:

- Must be generated by the extension before sending a push.
- Must stay the same across retries for the same logical push.
- Must be unique enough to avoid collisions, for example `arcana-${Date.now()}-${random}`.
- Arcana Web should treat repeated `pushId` values as idempotent retries where possible.

`channelId` requirements:

- Must be generated when the extension opens or reuses an Arcana window.
- Must remain stable for consecutive pushes to the same live Arcana window.
- Must change after the window is closed, replaced, or reopened after the extension lost the prior reference.
- Must be included in ping, ready, push, and ACK messages where available.
- The extension must ignore ACKs with the wrong `channelId`.

## Ping and Ready Messages

The extension must not rely on Arcana Web proactively messaging `window.opener`. Instead, readiness is discovered by extension-driven polling:

1. The extension opens or reuses the Arcana window.
2. The extension sends `arcana.ping` to the Arcana window every 500 ms while waiting for readiness.
3. Arcana Web replies with `arcana.ready` when storage is initialized and push handling is available.
4. The extension starts sending pushes only after receiving a valid `arcana.ready`, or after a push retry path explicitly decides to attempt sending despite missing readiness.

Ping shape:

```js
{
  _protocol: 1,
  _messageType: 'arcana.ping',
  channelId: 'session-...',
  sentAt: '2026-07-01T00:00:00.000Z',
  expectedOrigin: 'https://example.com'
}
```

Ping fields:

| Field | Required | Type | Meaning |
|---|---:|---|---|
| `channelId` | yes | string | Current window connection session ID. |
| `sentAt` | yes | ISO string | Ping send time. |
| `expectedOrigin` | recommended | string | Diagnostic value showing the origin the extension expects. |

Ready shape:

```js
{
  _protocol: 1,
  _messageType: 'arcana.ready',
  channelId: 'session-...',
  receiverId: 'string',
  supportedPushTypes: ['story.full', 'story.append', 'collection', 'regex'],
  readyAt: '2026-07-01T00:00:00.000Z'
}
```

Fields:

| Field | Required | Type | Meaning |
|---|---:|---|---|
| `receiverId` | yes | string | Arcana page instance ID. Useful for diagnostics. |
| `supportedPushTypes` | yes | string[] | Push types this page can save. |
| `readyAt` | yes | ISO string | Time when Arcana became ready. |
| `channelId` | yes | string | Echoes the ping `channelId`. |

Implementation note:

- Because Arcana may load after the extension opens it, readiness must be polling-based. The extension should keep pinging until it receives a valid ready reply, reaches the readiness timeout, or falls back.
- Arcana Web should respond to valid pings after storage is ready. If storage is not ready, it may ignore the ping or reply later; it should not send a false ready.

## Push Types

Phase 1 push types:

| `_pushType` | Purpose | Saves To |
|---|---|---|
| `story.full` | Save a full story archive. | `stories` |
| `story.append` | Append chapters/blocks to an existing story. | existing `stories` record |
| `collection` | Save one favorite/snippet/small theater item. | `collections` |
| `regex` | Save selected regex scripts. | `regex` |

Legacy compatibility:

- Existing `_pushType: 'full'` may be accepted as an alias for `story.full`.
- Existing `_pushType: 'append'` may be accepted as an alias for `story.append`.
- New extension code should emit the Phase 1 names above.

## Common Source Fields

These fields may appear on story, chapter, block, and collection payloads. Some fields may be optional in Phase 1, but the protocol reserves their names now.

| Field | Required in Phase 1 | Type | Meaning |
|---|---:|---|---|
| `sourceChatId` | recommended | string | SillyTavern current chat ID/file name from `context.chatId` or `getCurrentChatId()`. |
| `sourceCharacterId` | optional | string/number | SillyTavern character ID, if available. |
| `sourceCharacterName` | recommended | string | Current character name. |
| `sourceUserName` | optional | string | Current user name. |
| `sourceMessageId` | optional | string/number | Original SillyTavern message index or ID. |
| `sourceMessageIndex` | optional | number | Original SillyTavern message index. |
| `messageRange` | recommended for story pushes | object | Original message span covered by this push. |
| `pushId` | yes | string | Same ID as envelope pushId. |
| `blockId` | optional in Phase 1 | string | Stable ID for a saved text block. Reserved for future split storage and precise collection linking. |
| `updatedAt` | recommended | ISO string | Last meaningful update time for the payload. |

`messageRange` shape:

```js
{
  startIndex: 0,
  endIndex: 999,
  count: 1000,
  mode: 'full'
}
```

Fields:

| Field | Required | Type | Meaning |
|---|---:|---|---|
| `startIndex` | yes | number | First source chat message index included. |
| `endIndex` | yes | number | Last source chat message index included. |
| `count` | yes | number | Number of source messages considered or included. |
| `mode` | yes | string | `full`, `incremental`, or `selection`. |

`blockId` guidance:

- Phase 1 may omit `blockId`.
- Future versions should prefer deterministic IDs based on `sourceChatId`, source message index, and block index.
- Collection items should eventually point to `storyId` and `blockId` instead of only `source`.

## ACK Format

Arcana Web must return one ACK per push.

Success:

```js
{
  _protocol: 1,
  _messageType: 'arcana.ack',
  _pushType: 'story.full',
  pushId: 'arcana-...',
  channelId: 'session-...',
  ok: true,
  status: 'saved',
  saved: {
    storyId: 'story-id',
    collectionId: null,
    regexCount: 0,
    chapterCount: 3,
    blockCount: 200
  },
  receivedAt: '2026-07-01T00:00:00.000Z',
  savedAt: '2026-07-01T00:00:01.000Z'
}
```

Failure:

```js
{
  _protocol: 1,
  _messageType: 'arcana.ack',
  _pushType: 'story.append',
  pushId: 'arcana-...',
  channelId: 'session-...',
  ok: false,
  status: 'error',
  error: {
    code: 'STORY_NOT_FOUND',
    message: 'Story not found: story-id',
    retryable: false
  },
  receivedAt: '2026-07-01T00:00:00.000Z',
  failedAt: '2026-07-01T00:00:01.000Z'
}
```

ACK fields:

| Field | Required | Type | Meaning |
|---|---:|---|---|
| `_protocol` | yes | number | Must be `1`. |
| `_messageType` | yes | string | Must be `arcana.ack`. |
| `_pushType` | yes | string | Echoes the push type. |
| `pushId` | yes | string | Echoes the push ID. |
| `channelId` | yes | string | Echoes the channel ID. |
| `ok` | yes | boolean | Whether save succeeded. |
| `status` | yes | string | `saved`, `duplicate`, `partial`, or `error`. |
| `saved` | required when `ok: true` | object | Saved IDs/counts. |
| `error` | required when `ok: false` | object | Failure details. |
| `receivedAt` | yes | ISO string | When Arcana received the push. |
| `savedAt` | when success | ISO string | When save finished. |
| `failedAt` | when failure | ISO string | When failure was detected. |

Error codes:

| Code | Retryable | Meaning |
|---|---:|---|
| `INVALID_PROTOCOL` | false | `_protocol` missing or unsupported. |
| `INVALID_MESSAGE` | false | Required fields missing or malformed. |
| `UNSUPPORTED_PUSH_TYPE` | false | Arcana cannot handle this `_pushType`. |
| `STORAGE_NOT_READY` | true | IndexedDB/storage not initialized yet. |
| `STORAGE_FAILED` | true | IndexedDB write failed. |
| `STORY_NOT_FOUND` | false | Append target story does not exist. |
| `DUPLICATE_PUSH` | false | Same `pushId` was already saved. |
| `ORIGIN_REJECTED` | false | Message failed origin/source validation. |
| `UNKNOWN_ERROR` | true | Unexpected failure. |

Extension ACK recognition:

The extension must accept an ACK only if all checks pass:

- `event.data` is an object.
- `event.data._protocol === 1`.
- `event.data._messageType === 'arcana.ack'`.
- `event.data.pushId === pendingPush.pushId`.
- `event.data.channelId === pendingPush.channelId`.
- `event.origin === targetOrigin` for `http:` or `https:` Arcana URLs.
- `event.source === pushWindow` when the browser exposes the source.

If any check fails, the extension must ignore the message.

## Arcana Web Message Validation

Arcana Web must validate incoming protocol messages before saving data or replying.

For `arcana.ping`, Arcana Web should reply with `arcana.ready` only if all checks pass:

- `event.data` is an object.
- `event.data._protocol === 1`.
- `event.data._messageType === 'arcana.ping'`.
- `event.data.channelId` is a non-empty string.
- For `http:` or `https:` deployments, `event.origin` is present and accepted by the page's configured allow rules.
- Storage is initialized and ready to accept pushes.

For `arcana.push`, Arcana Web should process the message only if all checks pass:

- `event.data` is an object.
- `event.data._protocol === 1`.
- `event.data._messageType === 'arcana.push'`.
- `event.data._pushType` is one of the supported push types.
- `event.data.pushId` is a non-empty string.
- `event.data.channelId` is a non-empty string.
- For `http:` or `https:` deployments, `event.origin` is present and accepted by the page's configured allow rules.
- The payload contains the required fields for its `_pushType`.

If validation fails:

- Arcana Web must not save anything.
- Arcana Web should silently discard unrelated messages.
- Arcana Web may send an error ACK only when the message is clearly an Arcana protocol message with a valid `pushId` and `channelId`.
- Arcana Web must not rely on `event.source === window.opener` as a hard requirement, because `window.opener` may be `null` on mobile browsers.

## Timeout and Retry Strategy

Recommended Phase 1 timings:

- Ping interval while waiting for readiness: 500 ms.
- Ready timeout: 5000 ms.
- ACK timeout per attempt: 5000 ms.
- Retry count: 2 retries after the first attempt, for 3 total attempts.
- Delay between retries: 750 ms, then 1500 ms.

Flow:

1. Extension prepares payload and generates `pushId` and `channelId`.
2. Extension opens or reuses `window.open(archiveUrl, 'arcana-reader')`.
3. Extension sends `arcana.ping` every 500 ms until valid `arcana.ready` arrives or the ready timeout expires.
4. If readiness fails, extension may fall back or attempt one guarded push depending on implementation risk tolerance.
5. Extension sends the push via `postMessage`.
6. Extension waits for ACK.
7. If ACK success, extension may update `pushHistory`.
8. If ACK failure with `retryable: true`, retry until attempts are exhausted.
9. If ACK failure with `retryable: false`, stop retrying immediately and enter fallback behavior.
10. If no ACK before timeout, retry until attempts are exhausted.
11. If all attempts fail, use fallback behavior.

Fallback behavior:

- For `story.full`: download `.archive.json` and show a clear failure message.
- For `story.append`: download `.archive.json` append payload and do not update `pushHistory`.
- For `collection`: download `.archive.json` collection payload or queue locally in extension settings for later retry if implemented.
- For `regex`: show failure and allow user to retry manually.

Important rule:

- `pushHistory` must be written only after a success ACK. It must not be written immediately after `window.open` or after `postMessage`.

## Idempotency

Arcana Web should remember recently successful `pushId` values during the page lifetime.

Recommended behavior:

- If the same `pushId` arrives again and was already saved, return success ACK with `status: 'duplicate'`.
- If the same `pushId` arrives while still saving, return failure ACK with `STORAGE_NOT_READY` or queue internally.
- Phase 1 may keep idempotency memory in page memory only. Persistent idempotency can be added later.

## Large Payload Guidance

Phase 1 may send the complete payload in one `postMessage`.

Limits:

- URL length is no longer the limiting factor.
- Browser memory, structured clone cost, IndexedDB write time, and reader rendering remain limiting factors.

Future chunking is reserved but not required in Phase 1.

Reserved chunk message types:

- `arcana.push.begin`
- `arcana.push.chunk`
- `arcana.push.finish`
- `arcana.push.cancel`

Reserved chunk fields:

```js
{
  _protocol: 1,
  _messageType: 'arcana.push.chunk',
  _pushType: 'story.full',
  pushId: 'arcana-...',
  channelId: 'session-...',
  chunkIndex: 0,
  chunkCount: 10,
  chunkBytes: 524288,
  payloadEncoding: 'json',
  payloadChunk: {}
}
```

Chunking can be designed after the Phase 1 ACK path is stable.

## Backward Compatibility

Arcana Web should temporarily support:

- Existing `#push=` URL hash receiver.
- Existing `.archive.json` manual import.
- Legacy `_pushType: 'full'`.
- Legacy `_pushType: 'append'`.
- Existing `collection` and `regex` payloads without full source fields.

The extension should prefer the new protocol when possible, then fall back to legacy routes.

## Phase 1 Implementation Red Lines

These rules must be followed when implementation begins:

- Do not change `extractWithConfig` tag handling order.
- Keep the stable pipeline: protect keep -> strip/exclude -> restore.
- `pushHistory` write must wait for a success ACK.
- Do not write `pushHistory` after `window.open` alone.
- Clean the stale `goToChapterPaged` reference in `index.html` around line 1051 before delivery.
- After every JavaScript change, run `node --check` on the changed JS files before handing off.
- Pay special attention to mobile `postMessage` origin policy. Desktop success does not prove mobile success.

## Phase 1 Acceptance Checklist

- Pushing a full story opens or reuses only one Arcana tab named `arcana-reader`.
- Repeated collection pushes do not create multiple Arcana tabs.
- Extension updates `pushHistory` only after success ACK.
- If Arcana save fails, extension shows failure and does not mark the chat as pushed.
- If Arcana tab is closed, extension can reopen it and retry.
- If popup is blocked, extension falls back cleanly.
- Existing `#push=` and `.archive.json` paths still work.
- Legacy archives can still be imported.
- Mobile browser test verifies origin-safe message delivery or gracefully falls back.
